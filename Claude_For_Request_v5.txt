【三、全新视觉方向，强制要求】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

不得继续使用明显的 BoxGeometry、简单 ConeGeometry 和低面数几何体直接拼出主要画面的风格。基础几何体可以作为底层构件，但最终轮廓、材质、细节、光照和后期效果必须消除积木感。

1. 渲染技术选型
   · 启动时进行 WebGPU、WebGL2、浮点纹理、深度纹理、各向异性过滤和设备性能检测。
   · 支持时优先使用 WebGPU；否则使用经过完整优化的 WebGL2 渲染路径，不得因回退而退化为无波浪的静态平面。
   · 可以采用 Three.js WebGPURenderer / WebGLRenderer、TSL 或自定义 GLSL/WGSL、EffectComposer、postprocessing、three-mesh-bvh 等成熟方案。
   · 使用 ACES Filmic Tone Mapping、正确的 sRGB 输出、物理正确光照、PMREM 环境光照和合理曝光。
   · 所谓“光线追踪优化”必须按平台能力正确实现：WebGPU 高档可采用屏幕空间反射、BVH 辅助反射/阴影或有限路径追踪效果；WebGL2 使用高质量环境反射、SSR 近似、级联阴影、接触阴影和反射探针。不得虚假声称所有浏览器都启用了硬件光线追踪。
   · 提供 TAA/SMAA/FXAA 中至少一种稳定抗锯齿方案，解决细栏杆、桅杆、船体高光和远处海面闪烁。
   · 渲染技术必须服务于最终画面；仅引入库但没有实际应用，不视为完成。

2. 整体美术风格
   · 采用统一的半写实电影化航海风格。
   · 画面应拥有连续、圆润、流畅的轮廓。
   · 材质、雾效、光照、天空和水面必须属于同一视觉体系。
   · 避免过度鲜艳的塑料感、低多边形感和大量直角方块。
   · UI 采用现代海事仪表与科学实验界面融合的风格，但不得遮挡主要画面。

3. 船体建模与贴图
   · 使用程序化曲线、LatheGeometry、ExtrudeGeometry、ShapeGeometry、Catmull-Rom 曲线、细分曲面近似或自定义 BufferGeometry 创建流线型船体。
   · 船首、船尾、甲板边缘和船桥必须拥有圆滑连续的轮廓。
   · 使用平滑法线、足够细分段数和合理的 bevel，避免硬直方块轮廓。
   · 船体至少包含流线型外壳、甲板、船桥、曲面玻璃、栏杆、桅杆、雷达、导航灯、救生设备、真实船舵或舵机机构等细节。
   · 材质区分喷漆金属、湿润船壳、玻璃、木材、不锈钢和橡胶。
   · 湿润船体表面具有随视角变化的高光。
   · 船只随波面法线产生升沉、横摇和纵摇，而不是使用固定单一正弦动画。
   · 使用程序化 CanvasTexture、噪声贴图、细节法线、粗糙度贴图、金属度贴图和污渍遮罩增加尺度层次。
   · 正确设置 UV 或三平面映射，禁止出现明显拉伸、镜像错误、接缝跳变、摩尔纹和贴图游动。
   · 船体吃水线应有湿润区域、轻微盐渍和水线颜色变化，但不得用大面积重复纹理掩盖几何问题。
   · 船桥玻璃需要有受控反射、透明度和内部暗部，不能表现为纯黑块或完全透明空洞。
   · 阴影必须贴合船体，避免悬浮阴影、漏光、shadow acne 和 Peter Panning。

4. 海洋渲染，v4 核心验收项
   · 禁止用单张蓝色平面、简单透明材质或只有贴图滚动的平面冒充海洋。
   · 使用多层 Gerstner Waves、FFT 海洋频谱或同等级高质量自定义海洋 Shader。
   · 低幅长波负责轻微船体运动，细浪负责高光和法线细节。
   · 至少包含多个方向、不同波长和不同相速的波组，使水面无明显机械重复周期。
   · CPU 船体物理采样与 GPU 波面使用同一套波参数，保证船体确实浮在可见波面上。
   · 实现 Fresnel、天空反射近似、太阳高光路径、深浅色变化、距离雾化和地平线融合。
   · 使用程序化噪声或动态法线生成微小波纹。
   · v4 统一使用平静海况，不再提供常规、风浪或恶劣海况视觉预设。
   · 平静不等于静态平面：仍需保留连续长波、微小涟漪、细浪法线、柔和反射和轻微船体升沉。
   · 波高应明显低于 v3，避免大浪、密集白沫、强烈飞沫和剧烈横摇影响教学观察。
   · 船头与水面交界处只产生与船速匹配的轻柔破水和少量泡沫。
   · 船尾产生两条连续弯曲尾迹、螺旋桨扰流和逐渐消散的泡沫。
   · 船舶转向时尾迹轨迹必须随实际航迹弯曲。
   · 海平面不可出现明显边缘或网格边界。
   · 近景需要可辨识的水波起伏、法线细节和泡沫，中景体现方向性波列，远景自然融合至地平线。
   · 海面反射不能像金属镜面；粗糙度、Fresnel 和波面法线应共同决定反射强度。
   · 相机靠近水面时不得出现裁切、穿水、整片闪白或深度冲突。
   · 教学所需的侧风和洋流仍可作用于航向物理，但不得把海面切换成恶劣海况。

5. 统一多云天空和大气
   · 使用程序化天空 Shader、物理天空模型或 Three.js Sky。
   · 取消所有视觉时段选择按钮、日照时间滑块和晨曦/正午/夕阳切换逻辑。
   · 此处取消的是视觉时段与天气时间设置，不取消教学实验的 elapsed time、暂停、单步和倍速功能。
   · 全部模式统一使用白昼多云天气：清晰蓝天、自然白云、柔和白色日光和略带空气透视的地平线。
   · 云量保持中等，不得形成阴沉暴风天气，也不得完全遮蔽蓝天。
   · 天空亮度、太阳方向、曝光和色温保持固定，避免实验过程中光照变化干扰玩家船与参考船观察。
   · 云层使用多层程序化云、柔和透明体或 Shader 云，不得使用生硬球体堆叠。
   · 云层具有形变、明暗、远近视差和缓慢漂移。
   · 白云可以缓慢运动，但不得改变整体光照条件或被视作时间流逝系统。

6. 光照、阴影与后期处理
   · 可使用 EffectComposer。
   · 使用方向主光、天空/环境光、反射环境和局部导航灯形成一致照明。
   · 合理加入抗锯齿、Bloom、色调映射、颜色分级、轻微暗角、景深近似、太阳光晕、接触阴影和可选 SSR。
   · 后期效果必须克制，不能影响仪表和文字清晰度。
   · 提供低、中、高、超高画质档位。
   · 超高档可使用更高海面细分、更高粒子数量、更高阴影精度和更多后期效果。
   · Bloom 只作用于太阳、航行灯和高亮反射等合理区域，禁止全屏泛白。
   · 使用适合多云白昼的固定曝光和色调映射，不再实现时段曝光切换。
   · UI 必须在独立层保持清晰，不受景深、色差、运动模糊或 Bloom 污染。

7. 纯净无障碍海面
   · 删除岛屿、礁石、灯塔、浮标、航道门、其他船只、海鸟及所有可能出现在航线上的实体。
   · 禁止使用这些对象作为前景、中景、远景装饰或视觉参照。
   · 场景中允许存在的主要 3D 实体仅包括：玩家船、参考/伪影船、海面和天空云层。
   · 目标航向线、航迹线和教学标记可以保留，但必须是非碰撞、低干扰的辅助图形，不能表现为障碍物。
   · 船只不需要进行碰撞检测或避障；教学重点只放在航向控制与 PID 对比。

8. 性能
   · 目标桌面端 60 FPS。
   · 使用 InstancedMesh、对象池、几何体共享、材质共享、视锥裁剪和分级粒子数量。
   · 自动监测 FPS，并可建议或自动降低画质。
   · 即使不限制 HTML 大小，也不得因无意义堆叠对象造成严重卡顿。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【四、物理仿真与船舵反馈】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 使用固定物理步长 1/60 秒，并与 requestAnimationFrame 解耦。
2. 使用稳定的 Nomoto 航向模型或等效二阶模型：
   · dψ/dt = r
   · T·dr/dt + r = K·δ·v + τwind + τcurrent
3. 船舶具有偏航惯性、水动力阻尼和真实响应延迟。
4. 舵效与船速相关。
5. 最大实际舵角为 ±35°。
6. 区分：
   · 用户舵轮角度。
   · 控制器输出。
   · 指令舵角。
   · 液压舵机实际舵角。
7. 满舵约 1~3 秒到位。
8. 回舵速度可略快，但不得瞬时归零。
9. 回舵后船体继续偏航，随后在阻尼作用下减速。
10. 风扰包含缓慢风、随机阵风和可选恒定偏航力矩。
11. 洋流使航迹产生横向漂移，并在小地图上区分船头方向和地面航迹方向。
12. 航向误差使用最短角差，正确处理 359° 与 1°。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【五、永久可见的船舵，强制要求】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

只要当前允许学习者自行控制船舵，无论第一视角还是第三视角，都必须将船舵永久显示在画面中。

1. 船舵位置
   · v3 强制固定在页面水平中心，即主体船视觉中心的正下方；不得再放置于右下角。
   · 使用独立 HUD 层定位：left: 50%，通过 transform 居中，并为底部状态栏预留明确安全区域。
   · 舵轮中心应与页面水平中线对齐，而不是与右侧面板剩余空间对齐。
   · 不得遮挡底部状态栏、教学面板、小地图或主要航向视线。
   · 桌面端舵轮建议外径 280~360 px，至少比 v2 视觉尺寸增大 35%；移动端建议 190~240 px。
   · 屏幕尺寸较小时只能等比缩放和调整垂直安全间距，不能移到角落或隐藏。
   · 主体船、舵轮和下方数据区需要共同参与响应式布局，避免舵轮压住船尾、曲线或状态文字。

2. 船舵建模与表现
   · 优先使用精细 SVG 分层模型、Three.js 3D 模型或高质量 CSS/SVG 混合实现，不得继续使用粗糙圆环加矩形辐条。
   · 舵轮至少包含外轮缘、8 个圆滑握柄、辐条、中心轮毂、金属轴套和刻度环。
   · 使用连续曲线、圆角连接和平滑法线，确保握柄与轮缘连接自然。
   · 材质表现为经过保养的木材与黄铜/不锈钢，包含细微木纹、粗糙度差异、边缘高光、环境反射和接触阴影。
   · 舵轮具有层次明确的投影、内阴影、旋转惯性和液压阻尼反馈，但不能因过量 Bloom 失去细节。
   · 鼠标悬停、拖拽、键盘输入和到达机械限位时提供不同但克制的视觉反馈。
   · 支持鼠标和触摸拖拽。
   · A/D 或左右方向键控制时，舵轮同步旋转。
   · S 或空格回舵时，舵轮平滑返回。
   · 拖拽角度计算以舵轮真实中心为基准，跨越 ±180° 时不得突然跳变。

3. 舵角标注
   · 舵轮旁或轮毂下方必须始终明确显示当前角度，文字跟随居中布局但不随舵轮旋转。
   · 同时显示指令角与实际舵角，例如：
     “指令 +18.5°”
     “实际 +14.2°”
   · 标明 PORT / 左舷、MIDSHIPS / 中位、STARBOARD / 右舷。
   · 角度到达 ±35° 时显示限位提示。
   · 自动控制阶段如果不允许用户直接控制，舵轮仍可显示控制器动作，但应标注“自动舵接管”。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【六、状态变化因果指示系统，强制要求】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

状态栏不能只罗列数字。需要加入实时方向、趋势、因果关系和时间延迟提示，使学习者能够理解“控制变化 → 舵机变化 → 偏航变化 → 航向变化 → 误差变化”的链路。

1. 因果链显示
   · 在状态栏显式显示以下流程：
     误差 e → P/I/D 输出 → 指令舵角 → 实际舵角 → 偏航角速度 r → 当前航向 ψ
   · 每个环节使用箭头连接。
   · 当前变化最显著的环节高亮。
   · 使用颜色统一对应：
     - 误差：红色
     - P：橙色
     - I：紫色
     - D：青色
     - 指令舵角：黄色
     - 实际舵角：蓝色
     - 航向：绿色或白色

2. 趋势指示
   · 每个主要状态旁显示 ↑、↓、→ 或平滑趋势箭头。
   · 明确标注“正在增大”“正在减小”“基本稳定”。
   · 航向误差旁显示“正在接近目标”或“正在远离目标”。
   · 偏航角速度旁显示当前转向方向和转动趋势。
   · 指令舵角与实际舵角不同步时显示“舵机追随中”。

3. 延迟指示
   · 当用户或控制器刚改变指令，但船体尚未明显响应时，显示：
     “控制已施加，等待船体惯性响应”
   · 当舵已回中但船仍在转向时，显示：
     “舵已回中，偏航惯性仍在延续”
   · 当反向舵正在制动偏航时，显示：
     “反向舵正在降低偏航角速度”

4. P/I/D 解释提示
   · P 项变化时显示其与当前误差的直接对应关系。
   · I 项持续累积时显示累计方向及原因。
   · 当前误差接近零但 I 输出非零时，说明积分正在保持抵消扰动所需的舵量。
   · D 项与 P 项方向相反时，说明 D 正在提前制动。
   · 舵角饱和时显示 SATURATION。
   · Anti-windup 工作时显示积分暂停或回算状态。

5. 时间轴事件
   · 在曲线上标注目标阶跃、扰动加入、舵角饱和、误差过零、最大超调和进入稳定带。
   · 鼠标悬停事件时显示当时的误差、P/I/D、舵角和航向。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【七、教学模式课程】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 阶段 0：手动基线
   · 用户亲自掌舵。
   · 先体验无风目标阶跃，再加入恒定侧风。
   · 记录平均绝对误差、最大误差、操作次数、误差过零次数和控制能量。

2. 阶段 1：P
   · uP = Kp·e
   · δcmd = clamp(uP, ±35°)
   · 提供 Kp 滑块和过小、平衡、过大预设。
   · 展示 Kp 过小响应慢、Kp 过大可能摆动。
   · 在持续扰动下突出稳态误差、P 输出和抵消扰动所需舵量。
   · 严谨说明：并非所有 P 控制系统必然有稳态误差；在本模型的恒定扰动条件下，P 需要非零误差产生持续补偿舵量。

3. 阶段 2：PI
   · uP = Kp·e
   · uI = Ki·∫e dt
   · δcmd = clamp(uP + uI, ±35°)
   · 使用与 P 完全相同的扰动。
   · 使用积分容器或记忆槽显示误差累计。
   · 说明误差接近零后 I 项仍能维持抵消扰动所需的舵量。
   · 展示 Ki 过大导致的超调、振荡和积分饱和。
   · 实现可切换 Anti-windup。

4. 阶段 3：PID
   · uP = Kp·e
   · uI = Ki·∫e dt
   · uD = Kd·de/dt，或使用测量值微分。
   · δcmd = clamp(uP + uI + uD, ±35°)
   · D 项必须经过可调低通滤波。
   · 可切换显示原始微分和滤波微分。
   · 展示 D 项如何降低超调、振荡和调节时间。
   · 展示 Kd 过大造成的噪声敏感和舵机抖动。

5. 每个阶段包含：
   · 实验前预测问题。
   · 当前目标。
   · 实时提示。
   · 参数调节。
   · 开始、暂停、单步、倍速和重置。
   · 结果指标。
   · 正确解释。
   · 常见误区。
   · 下一阶段入口。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【八、教学引导模式、最优参考船与记录伪影，最高优先级】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

教学模式中加入可随时开启或关闭的“引导模式 Guided Mode”。v2 中只有参数卡片、透明材质或并排演示而没有真正双实体同步运行的实现，均视为未完成。

0. v3 强制完成定义
   · 画面中必须同时存在两个可辨识个体：学习者当前控制的实体船，以及最优/最快有效记录的半透明伪影船。
   · 两艘船必须在同一个 3D 世界、同一实验时钟和同一确定性场景中同时运动，而不是轮流播放、只画两条曲线或只在小地图显示图标。
   · 伪影船必须拥有可见船体、姿态、舵角、尾迹或轨迹、名称标签和实时差距。
   · 暂停、继续、单步、倍速、重置和拖动实验时间轴时，实体船与伪影船必须保持时间同步。
   · 禁止让伪影直接复制当前船的位置，也禁止每帧瞬移至预设点；必须使用固定步长状态或带插值的记录轨迹平滑重放。
   · 当没有历史记录时，系统必须自动生成并保存一条当前场景下的参考最优记录，不能让伪影功能处于空白状态。

1. 引导模式目标
   · 帮助学习者理解自己当前参数与推荐参数的区别。
   · 推荐参数不是固定常数，而应根据当前船速、风力、洋流、目标阶跃大小和船舶模型动态选择。
   · 推荐值应来自预先离线搜索、运行时小规模参数搜索、规则调度 Gain Scheduling 或可靠的参数表插值。
   · 不得使用完全随机或没有依据的所谓“最优参数”。

2. 推荐参数显示
   · 实时显示：
     - 当前用户 Kp、Ki、Kd
     - 当前推荐 Kp*、Ki*、Kd*
     - 参数差值 ΔKp、ΔKi、ΔKd
   · 对每个差值给出简短解释，例如：
     “你的 Kp 偏大，响应更快，但超调风险更高。”
     “你的 Ki 偏小，持续扰动下消除静差需要更长时间。”
     “你的 Kd 偏小，接近目标时制动不足。”
   · 显示推荐参数所优化的目标：
     - 最小 IAE
     - 最小超调
     - 最短调节时间
     - 平衡模式
   · 允许用户切换优化目标，推荐参数随目标更新。

3. 参数搜索方法
   · 推荐在独立的简化物理副本中并行模拟一组候选参数。
   · 候选参数使用与玩家完全相同的初始状态和确定性扰动。
   · 使用综合代价函数，例如：
     J = w1·IAE + w2·Overshoot + w3·SettlingTime + w4·ControlEnergy
   · 不同优化目标使用不同权重。
   · 参数搜索不能阻塞主渲染循环。
   · 可以分帧计算、使用 Web Worker、预计算表格或在环境变化后低频重新计算。
   · 明确将结果称为“当前场景下的推荐参数”或“参考最优参数”，不要声称是所有系统的全局最优解。

4. 半透明最优参考船
   · 在玩家船旁或相同起点叠加一艘半透明参考船。
   · 参考船使用推荐 PID 参数，运行独立但相同的船舶物理副本。
   · 参考船与玩家船拥有：
     - 相同初始航向
     - 相同目标航向
     - 相同船速
     - 相同确定性风力
     - 相同洋流
     - 相同舵机约束
   · 参考船材质半透明，建议使用青绿色或淡金色轮廓。
   · 参考船具有柔和发光边缘或轮廓线，但不能遮挡玩家船。
   · 参考船旁显示“参考 PID”标签。
   · 玩家船与参考船的航迹使用不同颜色。
   · 小地图同时显示两艘船的航迹。
   · 实时显示双方：
     - 航向误差
     - 舵角
     - 偏航角速度
     - IAE
     - 超调
     - 调节状态

5. 空间显示方式
   · 默认参考船与玩家船从同一起点出发，但为了避免完全重叠，可采用轻微横向视觉偏移。
   · 该偏移只能用于显示，不得改变参考船的物理航迹和误差计算。
   · 提供“重合幽灵”“左右并排”“仅显示航迹”三种查看方式。

6. 自行控制时的引导
   · 在手动控制阶段，引导模式可显示一艘由参考 PID 控制的半透明船。
   · 学习者可比较自己的转舵时机与参考 PID 的舵角变化。
   · 永久船舵旁同时显示：
     - 我的舵角
     - 参考舵角
   · 当用户明显回舵过晚、舵量过大或反向制动不足时，给出提示，但不得强制替用户操作。

7. 自动控制参数实验时的引导
   · 玩家船使用学习者设定的 PID 参数。
   · 参考船使用推荐 PID 参数。
   · 两艘船同时运行。
   · 参数变化后，可选择立即重置对比实验，避免不同历史状态造成不公平比较。
   · 引导面板需要说明当前差异来自哪个控制分量。

8. 引导模式 UI
   · 提供清晰的开关。
   · 提供推荐参数卡片。
   · 提供优化目标选择。
   · 提供参考船显示方式选择。
   · 提供“应用推荐参数”按钮。
   · 提供“恢复我的参数”按钮。
   · 应用推荐参数前保存当前用户参数，方便对比和撤销。

9. 竞速游戏式记录伪影
   · 参考赛车、竞速和刷记录游戏的 Ghost Replay 机制：本次运行是实体主体，最佳有效运行是不可碰撞的半透明伪影。
   · 提供“系统最优”“我的最佳”“隐藏伪影”三个选项。
   · “系统最优”来自当前固定场景下的推荐 PID 或经过搜索得到的合法参考轨迹。
   · “我的最佳”来自用户在同一场景标识、模型版本、扰动种子、船速和目标航向条件下完成的最佳有效记录。
   · 伪影不参与碰撞、不影响风浪、不改变玩家物理，只负责视觉和教学对比。
   · 伪影材质使用半透明青蓝色、轻微边缘发光、深度衰减和抖动透明，避免传统透明排序造成船体部件显示错乱。
   · 当两船重合时，仍可通过轮廓线、标签、轻微时间残影或可切换视觉横向偏移辨识双方。

10. “最快/最优”的严谨定义
   · PID 教学中的最快记录不能仅以首次穿过目标航向为准，否则会奖励严重超调和振荡。
   · 默认“最快有效记录”定义为：进入目标容差带并连续保持规定时间后的最短调节时间。
   · 记录有效性至少要求：最大超调不超过阈值、舵角未非法越界、实验全程未暂停作弊、最终误差进入稳定带。
   · 同时保存 Settling Time、Overshoot、IAE、Control Energy 和最大舵角。
   · 若两个记录调节时间相同，则依次以更小超调、更小 IAE、更低控制能量决定排名。
   · 允许切换“最快稳定”“最小 IAE”“最小超调”“综合最优”排行榜，但必须清楚显示当前评价标准。

11. 伪影记录数据
   · 以固定 1/60 秒或明确的固定采样率记录时间戳、位置、航向、横摇、纵摇、升沉、偏航角速度、指令舵角、实际舵角、P/I/D 分量和关键事件。
   · 记录场景 ID、随机种子、物理参数版本、控制器类型、PID 参数和评价指标，防止不同场景记录被错误比较。
   · 重放时使用线性位置插值和最短角度/四元数插值，避免卡顿、绕远旋转和跨 360° 跳变。
   · 页面刷新后使用 localStorage 或 IndexedDB 保留个人最佳记录；数据损坏或版本不兼容时安全回退到系统参考记录。
   · 提供清除个人记录按钮，并在执行前明确确认。

12. 实时差距表达
   · 伪影标签显示“系统最优”或“我的最佳”，实体船标签显示“当前运行”。
   · HUD 实时显示时间差 Δt、航向误差差 Δe、舵角差 Δδ、IAE 差和当前领先/落后状态。
   · 时间差应按双方达到相同航向进度或稳定阶段的时间估计，不能简单用两者世界坐标距离代替。
   · 关键时刻显示“参考船已开始回舵”“你比参考晚回舵 0.8 s”“参考船已进入稳定带”等教学提示。
   · 曲线、小地图和 3D 航迹必须使用一致颜色区分当前运行与伪影。

13. 完整性测试
   · 开始实验后立即能看到两艘船。
   · 两船在相同扰动下产生可比较但独立的运动。
   · 改变用户参数只影响当前实体船，不能污染已保存伪影。
   · 切换伪影来源后正确从 t=0 重置并同步。
   · 伪影重放到记录末尾后保持最终状态或明确淡出，不得消失报错。
   · WebGL 透明排序、深度写入和轮廓后期处理不得导致伪影整船缺面、闪烁或遮住实体船。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【九、控制器与状态可视化】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 分量合成图
   · P 使用橙色。
   · I 使用紫色。
   · D 使用青色。
   · 总输出使用白色。
   · 饱和后指令使用黄色。
   · 实际舵角使用蓝色。
   · 正值指向右舷，负值指向左舷。
   · 上述分量色用于瞬时控制量合成图；教学状态曲线必须改用下述“玩家/参考主体色系”，两者不能混淆。

2. 玩家船与参考船分别展示
   · 教学状态栏必须同时包含玩家船和参考船的完整状态曲线，不能只显示玩家曲线加一条参考航向。
   · 两个主体至少分别展示：
     - 当前航向 ψ
     - 航向误差 e
     - P 输出
     - I 输出
     - D 输出
     - 指令舵角 δcmd
     - 实际舵角 δactual
     - 偏航角速度 r
   · 目标航向 ψref 作为中性辅助线，只绘制一次，不归属于任一主体。
   · 玩家和参考的同名变量必须在同一时间轴和同一量纲轨道内成对绘制，方便直接对比。

3. 主体色系规则，v4 强制要求
   · 玩家船全部状态曲线统一采用蓝色/青蓝色系。
   · 参考船或最优伪影全部状态曲线统一采用橙色/金色系。
   · 不得继续让航向、误差、P、I、D、舵角分别使用彼此无关的彩虹颜色。
   · 同一主体内的不同变量使用同一色相家族的明度、饱和度、透明度变化，并结合线型区分：
     - 航向：粗实线
     - 误差：中等虚线
     - 指令舵角：点划线
     - 实际舵角：细实线
     - P/I/D：使用不同短划线、点线或标记
     - 偏航角速度：细点线
   · 玩家与参考的同名变量使用相同线型，只通过蓝青/橙金色系区分主体。
   · 图例首先按“我的船”和“参考船”分组，再列出组内变量；不得把所有曲线混成一排难以辨认的图例。
   · 除颜色外必须使用线型、组标题、悬停标签和可选标记保证色觉障碍用户也能区分。

4. 图表分轨与零值观察
   · 不应把航向、误差、舵角、P/I/D 和偏航角速度全部强行压缩在同一个 Y 轴。
   · 使用共享时间轴的同步分轨图，建议至少分为：
     - 航向轨：目标航向、玩家航向、参考航向
     - 误差轨：玩家误差、参考误差
     - 控制轨：双方 P/I/D、指令舵角和实际舵角
     - 偏航率轨：双方偏航角速度
   · 各轨道可折叠，但教学模式默认展开“航向轨”和“误差轨”。
   · 鼠标悬停任一时间点时，所有轨道显示同步垂直游标和双方数值。

5. Y 轴刻度与零线
   · 相比 v3 明显减小 Y 轴主刻度距离，并增加浅色次网格线。
   · 误差轨默认主刻度间隔不大于 2°；当误差进入 ±5° 时自动细化到 1°，进入 ±2° 时可细化到 0.5°。
   · 舵角/控制轨默认主刻度间隔不大于 5°。
   · 偏航角速度轨根据当前范围使用 0.1°/s、0.2°/s 或 0.5°/s 等合理细刻度。
   · 误差轨必须始终绘制清晰的 0° 基准线，并将 ±0.5°、±1° 或当前稳定带以轻微背景带标出。
   · 自动缩放必须保留 0 值，不得因数据范围偏向一侧而把零线裁出画面。
   · 提供“零点放大”开关：开启后优先显示误差 ±5° 或更小范围，便于判断误差是否真正趋近 0。
   · 刻度标签保留足够小数位，例如 0.5°、1.0°，避免取整掩盖稳态误差。

6. 状态图尺寸
   · 状态曲线区域在桌面教学模式下应占页面高度约 32%~42%，相较 v3 明显增大。
   · 图表绘图区不能因图例、标题或按钮挤压而实际变小。
   · 可适度压缩普通数字状态卡和说明区，为曲线留出更大的连续观察面积。
   · 1366×768 分辨率下，误差轨有效绘图高度不得低于 90 px；1920×1080 下不得低于 130 px。
   · 移动端允许分页或全屏展开状态图，但不能把所有轨道压缩成无法辨认的细条。

7. 实时图表支持开关
   · 目标航向
   · 我的船全部变量
   · 参考船全部变量
   · 航向轨
   · 误差轨
   · 控制轨
   · 偏航率轨
   · 零点放大
   · 允许进一步展开选择单个变量，但主体总开关必须始终存在。

8. 对比指标
   · Rise Time
   · Overshoot
   · Settling Time
   · Steady-State Error
   · IAE
   · Control Energy

9. 图表必须支持
   · 最近 60 秒滚动。
   · 教学固定时长实验。
   · 事件标记。
   · 暂停后悬停查看数据。
   · P、PI、PID 历史曲线叠加。
   · 玩家船与参考船实时曲线叠加。
   · 同步游标、零点放大和细粒度刻度切换。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【十、界面重新设计与防重叠要求，强制要求】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 使用明确的布局系统
   · 使用 CSS Grid 和 Flexbox。
   · 不要依赖大量固定绝对坐标堆叠所有面板。
   · 主视口、顶部栏、侧栏、舵轮区和底部数据区必须拥有明确安全区域。
   · 所有浮层设置合理 z-index 层级。

2. 桌面布局建议
   · 顶部：模式、视角和全局控制；不得出现时段或时间天气切换。
   · 左侧：教学步骤或挑战任务。
   · 右侧：参数、引导模式和推荐参数。
   · 底部：放大的多轨响应图、因果链和精简状态数据。
   · 教学模式优先保证状态曲线面积，底部区域建议占页面高度 32%~42%。
   · 主视口中央尽量保持无遮挡。

3. 防止文字重叠
   · 所有文本容器必须设置合理 min-width、max-width、line-height 和 overflow 规则。
   · 禁止把重要文字依赖 text-overflow: ellipsis 隐藏。
   · 长文本应自动换行或进入可滚动区域。
   · 数值使用等宽字体并预留最大字符宽度。
   · 按钮文字不得挤压到两行后覆盖相邻控件。
   · 中英文混排时保持足够行高。
   · 任何分辨率下不得出现文字相互覆盖。

4. 响应式断点
   · 至少检查 1920×1080、1440×900、1366×768、1024×768、768×1024 和 390×844。
   · 窄屏下侧栏改为抽屉或分页。
   · 底部图表可以折叠。
   · 舵轮缩放但保持永久可见。
   · 小地图、参数面板和教学卡片不得互相覆盖。

5. 可访问性
   · 所有按钮有清晰标签和焦点样式。
   · 颜色之外同时使用文字、图标或线型表达含义。
   · 提供减少镜头晃动和静音选项。
   · 主要文字保证足够对比度。

6. 运行时布局检测
   · 可在开发代码中加入可关闭的 UI 边界检查。
   · 使用 ResizeObserver 在布局变化时重新计算 Canvas 和图表尺寸。
   · 模式切换、窗口缩放和面板展开后必须重新布局。

【十一、视角与交互】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 第三人称
   · 摄像机位于船后上方并平滑跟随。
   · 鼠标右键环绕。
   · 滚轮有限缩放。
   · 镜头不得穿入船体或海面。

2. 第一人称
   · 位于船桥或甲板前方。
   · 能看到流线型船头、船桥前景和海面。
   · 船体运动反馈克制且连贯。

3. 视角切换
   · 使用 0.5~0.8 秒平滑过渡。
   · 不得瞬移或突然改变方向。

4. 键盘
   · A/D 或左右方向键转舵。
   · S 或空格回舵。
   · V 切换视角。
   · P 暂停教学实验。
   · R 重置。

5. 触控
   · 支持拖拽舵轮。
   · 主要控件有足够点击区域。

【十三、HTML 与原生 EXE 技术路线】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 默认路线：HTML
   · 首先尝试 WebGPU，回退到 WebGL2，并通过实际运行效果决定渲染路径。
   · HTML 方案应保持打开即用，可通过本地静态服务器运行。
   · 使用视觉扩展库时必须锁定兼容版本，避免 CDN 模块版本混用。
   · 需要提供加载进度、着色器编译失败提示、WebGPU/WebGL2 能力提示和画质自动降级。

2. 允许切换为 EXE 的条件
   · 只有在浏览器方案经验证仍无法同时满足海洋效果、双船伪影、船体质量和目标帧率时，才切换到原生 EXE。
   · 切换前应明确记录 HTML 方案无法通过的验收项，不得仅以“实现困难”为理由放弃浏览器方案。
   · 原生版本必须完整保留体验模式、Manual/P/PI/PID 教学、固定种子实验、双船伪影、曲线、指标、大型居中船舵和响应式/可缩放 UI。

3. C++ EXE 路线
   · 对高质量实时海洋、PBR、GPU 粒子和稳定性能有明确需求时，优先选择 C++。
   · 可采用 SDL2/GLFW + OpenGL/Vulkan/DirectX，或选择能够合法打包的成熟 C++ 渲染/游戏框架。
   · 物理与渲染必须解耦，记录伪影仍使用固定时间步和结构化轨迹数据。
   · 构建应为 Release，提供可重复构建脚本和明确依赖版本。

4. Python EXE 路线
   · 若开发效率优先且目标机器性能足够，可使用 Python + ModernGL/Panda3D/PySide6 等适合 3D 与 UI 的技术组合。
   · 使用 PyInstaller 或等效工具打包时，必须验证资源路径、Shader、动态库和首次启动。
   · 不得使用低质量 2D 占位画面替代原本要求的 3D 海洋和双船伪影。

5. EXE 交付要求
   · 目标平台默认 Windows 10/11 x64。
   · 提供可直接运行的 EXE、源代码、构建说明和依赖许可证说明。
   · 程序不得要求用户另外安装 Python、编译器或开发环境。
   · 首次运行、无独显、窗口缩放、全屏切换和退出重启均需验证。
   · 用户记录保存在明确的用户数据目录，升级版本时不得静默损坏或覆盖。

【十五、验收标准】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

视觉：
✓ 主要船体和环境不再具有明显方块拼接感。
✓ 船体轮廓流线、平滑且细节丰富。
✓ 船体不存在明显贴图拉伸、接缝、闪烁、漏光、阴影悬浮和不合理镜面高光。
✓ 海洋不是静态蓝色平面，近景可清楚看到连续水波和海浪。
✓ 海洋保持平静，具有低幅长波、细小涟漪、Fresnel、柔和反射、高光、轻微破水和连续尾迹。
✓ 波浪视觉与船体轻微升沉、横摇、纵摇及浮力采样一致，不出现恶劣海况。
✓ 所有岛屿、礁石、灯塔、浮标、航道门、其他船只、海鸟和障碍物均已移除。
✓ 场景除玩家船、参考船、海面、天空云层和非碰撞辅助线外没有其他实体。
✓ 全部模式统一为固定多云白昼，拥有蓝天白云、柔和日光和稳定曝光。
✓ 界面中不存在晨曦、正午、夕阳或其他时段切换控件。
✓ 后期效果提升画面但不影响 UI。
✓ WebGPU 和 WebGL2 路径均有合理能力检测与回退，不虚假标注光线追踪。

界面：
✓ 所有指定分辨率下无文字重叠、控件覆盖和面板越界。
✓ 主视口、教学区、参数区、图表和状态区层级清晰。
✓ 文本可读，不依赖截断隐藏关键信息。
✓ 模式和面板切换后布局正确。
✓ 教学状态曲线区域相较 v3 明显增大，并满足规定的有效绘图高度。

状态曲线：
✓ 玩家船的全部曲线使用蓝色/青蓝色系。
✓ 参考船的全部曲线使用橙色/金色系。
✓ 同一变量在两个主体中使用相同线型，并通过主体色系区分。
✓ 目标航向使用中性辅助色，不破坏两种主体色系。
