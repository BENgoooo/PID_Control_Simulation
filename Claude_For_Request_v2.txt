【v2 融合后的总 Prompt】

请重新设计并完整实现一个基于 WebGL 的网页版“船舵操控与 PID 教学模拟器”。最终交付物为一个可直接运行的独立 HTML 文件，允许内嵌任意规模的 CSS、JavaScript、Shader、SVG、CanvasTexture、程序化音频及数据，不必考虑 HTML 文件大小。

可以通过 CDN 使用 Three.js 稳定版及其官方扩展模块，也可以使用其他适合的 WebGL、后期处理、动画、图表、物理或 UI 库。不得使用需要额外下载的本地资源文件；图片、纹理、模型和音效应尽量通过程序化生成、内嵌 Data URI、内嵌 Base64 或代码构建，保证最终仍然只有一个 HTML 文件。

本项目不是普通的轮船驾驶小游戏，而是一套以水手手动掌舵为物理原型、以可重复实验为核心的 PID 交互学习工具。作品必须同时提供高完成度的沉浸驾驶体验和严谨、直观、可验证的 PID 教学体验。

【一、最高优先级目标】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 彻底消除现有版本的“方块感”和基础几何体拼接感。
2. 采用新的视觉技术、建模方式、材质风格和后期处理，使海洋、船体、天空、光照、环境和镜头达到流畅、精美、统一的高质量网页 3D 游戏观感。
3. 保留真实可感知的船舶航向动力学、偏航惯性、舵机延迟、风浪扰动和洋流漂移。
4. 提供体验模式与教学模式。
5. 教学模式按照 Manual → P → PI → PID 逐步展开，并提供可选“引导模式”。
6. 让学习者同时看见：
   · 控制器为什么产生当前控制量。
   · 舵角如何变化。
   · 船舶状态为何随后变化。
   · 自己的控制结果与推荐 PID 参数结果存在什么区别。
7. 所有 UI 必须经过完整布局检查，不允许文字重叠、控件遮挡、文字溢出、图表覆盖、面板越界等低级问题。

【二、双模式结构】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

首次进入页面时显示精致的模式选择界面，并允许在运行过程中切换。

1. 体验模式 Experience
   · 用户完全手动掌舵。
   · 重点提升画面、操控手感、镜头反馈、声效和沉浸感。
   · 提供自由航行、航向保持、阵风、横流和航道挑战。
   · PID 自动控制默认关闭，但仍显示误差、误差积分和变化趋势，帮助用户形成控制直觉。

2. 教学模式 Tutorial
   · 依次包含手动基线、P、PI、PID 四个阶段。
   · 每个阶段使用相同随机种子、相同目标航向、相同侧风、相同洋流和相同初始状态。
   · 支持参数调节、暂停、继续、单步、重置、时间倍速、保存实验、曲线叠加和指标对比。
   · 提供“自由实验”和“引导模式”两种教学方式。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【三、全新视觉方向，强制要求】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

不得继续使用明显的 BoxGeometry、简单 ConeGeometry 和低面数几何体直接拼出主要画面的风格。基础几何体可以作为底层构件，但最终轮廓、材质、细节、光照和后期效果必须消除积木感。

1. 整体美术风格
   · 采用统一的半写实电影化航海风格。
   · 画面应拥有连续、圆润、流畅的轮廓。
   · 材质、雾效、光照、天空和水面必须属于同一视觉体系。
   · 避免过度鲜艳的塑料感、低多边形感和大量直角方块。
   · UI 采用现代海事仪表与科学实验界面融合的风格，但不得遮挡主要画面。

2. 船体建模
   · 使用程序化曲线、LatheGeometry、ExtrudeGeometry、ShapeGeometry、Catmull-Rom 曲线、细分曲面近似或自定义 BufferGeometry 创建流线型船体。
   · 船首、船尾、甲板边缘和船桥必须拥有圆滑连续的轮廓。
   · 使用平滑法线、足够细分段数和合理的 bevel，避免硬直方块轮廓。
   · 船体至少包含流线型外壳、甲板、船桥、曲面玻璃、栏杆、桅杆、雷达、导航灯、救生设备、真实船舵或舵机机构等细节。
   · 材质区分喷漆金属、湿润船壳、玻璃、木材、不锈钢和橡胶。
   · 湿润船体表面具有随视角变化的高光。
   · 船只随波面法线产生升沉、横摇和纵摇，而不是使用固定单一正弦动画。

3. 海洋渲染
   · 使用多层 Gerstner Waves、FFT 风格近似或高质量自定义海洋 Shader。
   · 大浪负责船体运动，细浪负责高光和法线细节。
   · 实现 Fresnel、天空反射近似、太阳高光路径、深浅色变化、距离雾化和地平线融合。
   · 使用程序化噪声或动态法线生成微小波纹。
   · 风力增大时，波高、波长、传播速度、白沫和飞沫同步增强。
   · 船头与水面交界处产生破浪泡沫和水花。
   · 船尾产生两条连续弯曲尾迹、螺旋桨扰流和逐渐消散的泡沫。
   · 船舶转向时尾迹轨迹必须随实际航迹弯曲。
   · 海平面不可出现明显边缘或网格边界。

4. 天空和大气
   · 使用程序化天空 Shader、物理天空模型或 Three.js Sky。
   · 支持晨曦、正午、夕阳三个明显不同的时段。
   · 晨曦包含低位暖色太阳、橙粉大气、薄雾和柔和倒影。
   · 正午包含高位太阳、深蓝天空、清晰云层和高对比海面。
   · 夕阳包含金橙紫渐变、长距离暖色反射、空气透视和太阳光晕。
   · 云层使用多层程序化云、柔和透明体或 Shader 云，不得使用生硬球体堆叠。
   · 云层具有形变、明暗、远近视差和缓慢漂移。

5. 后期处理
   · 可使用 EffectComposer。
   · 合理加入抗锯齿、Bloom、色调映射、颜色分级、轻微暗角、景深近似和太阳光晕。
   · 后期效果必须克制，不能影响仪表和文字清晰度。
   · 提供低、中、高、超高画质档位。
   · 超高档可使用更高海面细分、更高粒子数量、更高阴影精度和更多后期效果。

6. 环境细节
   · 使用流线型或远景剪影形式创建岛屿、灯塔、浮标、其他船只和海鸟。
   · 浮标随波浪运动，并可作为航道挑战参照。
   · 岛屿使用噪声地形或平滑轮廓，不得使用单个低面数圆锥。
   · 场景元素按前景、中景和远景组织，并使用空气透视形成层次。

7. 性能
   · 目标桌面端 60 FPS。
   · 使用 InstancedMesh、对象池、几何体共享、材质共享、视锥裁剪和分级粒子数量。
   · 自动监测 FPS，并可建议或自动降低画质。
   · 即使不限制 HTML 大小，也不得因无意义堆叠对象造成严重卡顿。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【四、物理仿真与船舵反馈】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 使用固定物理步长 1/60 秒，并与 requestAnimationFrame 解耦。
2. 使用稳定的 Nomoto 航向模型或等效二阶模型：
   · dψ/dt = r
   · T·dr/dt + r = K·δ·v + τwind + τcurrent
3. 船舶具有偏航惯性、水动力阻尼和真实响应延迟。
4. 舵效与船速相关。
5. 最大实际舵角为 ±35°。
6. 区分：
   · 用户舵轮角度。
   · 控制器输出。
   · 指令舵角。
   · 液压舵机实际舵角。
7. 满舵约 1~3 秒到位。
8. 回舵速度可略快，但不得瞬时归零。
9. 回舵后船体继续偏航，随后在阻尼作用下减速。
10. 风扰包含缓慢风、随机阵风和可选恒定偏航力矩。
11. 洋流使航迹产生横向漂移，并在小地图上区分船头方向和地面航迹方向。
12. 航向误差使用最短角差，正确处理 359° 与 1°。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【五、永久可见的船舵，强制要求】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

只要当前允许学习者自行控制船舵，无论第一视角还是第三视角，都必须将船舵永久显示在画面中。

1. 船舵位置
   · 默认固定在主视口底部中央或右下安全区域。
   · 不得遮挡底部状态栏、教学面板、小地图或主要航向视线。
   · 屏幕尺寸较小时自动缩放或移动，但不能隐藏。

2. 船舵表现
   · 使用精细 SVG、Canvas 2D、Three.js 3D 舵轮或高质量 CSS/SVG 混合实现。
   · 舵轮具有圆润轮廓、木材或金属材质感、阴影和旋转反馈。
   · 支持鼠标和触摸拖拽。
   · A/D 或左右方向键控制时，舵轮同步旋转。
   · S 或空格回舵时，舵轮平滑返回。

3. 舵角标注
   · 舵轮旁必须始终明确显示当前角度。
   · 同时显示指令角与实际舵角，例如：
     “指令 +18.5°”
     “实际 +14.2°”
   · 标明 PORT / 左舷、MIDSHIPS / 中位、STARBOARD / 右舷。
   · 角度到达 ±35° 时显示限位提示。
   · 自动控制阶段如果不允许用户直接控制，舵轮仍可显示控制器动作，但应标注“自动舵接管”。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【六、状态变化因果指示系统，强制要求】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

状态栏不能只罗列数字。需要加入实时方向、趋势、因果关系和时间延迟提示，使学习者能够理解“控制变化 → 舵机变化 → 偏航变化 → 航向变化 → 误差变化”的链路。

1. 因果链显示
   · 在状态栏显式显示以下流程：
     误差 e → P/I/D 输出 → 指令舵角 → 实际舵角 → 偏航角速度 r → 当前航向 ψ
   · 每个环节使用箭头连接。
   · 当前变化最显著的环节高亮。
   · 使用颜色统一对应：
     - 误差：红色
     - P：橙色
     - I：紫色
     - D：青色
     - 指令舵角：黄色
     - 实际舵角：蓝色
     - 航向：绿色或白色

2. 趋势指示
   · 每个主要状态旁显示 ↑、↓、→ 或平滑趋势箭头。
   · 明确标注“正在增大”“正在减小”“基本稳定”。
   · 航向误差旁显示“正在接近目标”或“正在远离目标”。
   · 偏航角速度旁显示当前转向方向和转动趋势。
   · 指令舵角与实际舵角不同步时显示“舵机追随中”。

3. 延迟指示
   · 当用户或控制器刚改变指令，但船体尚未明显响应时，显示：
     “控制已施加，等待船体惯性响应”
   · 当舵已回中但船仍在转向时，显示：
     “舵已回中，偏航惯性仍在延续”
   · 当反向舵正在制动偏航时，显示：
     “反向舵正在降低偏航角速度”

4. P/I/D 解释提示
   · P 项变化时显示其与当前误差的直接对应关系。
   · I 项持续累积时显示累计方向及原因。
   · 当前误差接近零但 I 输出非零时，说明积分正在保持抵消扰动所需的舵量。
   · D 项与 P 项方向相反时，说明 D 正在提前制动。
   · 舵角饱和时显示 SATURATION。
   · Anti-windup 工作时显示积分暂停或回算状态。

5. 时间轴事件
   · 在曲线上标注目标阶跃、扰动加入、舵角饱和、误差过零、最大超调和进入稳定带。
   · 鼠标悬停事件时显示当时的误差、P/I/D、舵角和航向。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【七、教学模式课程】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 阶段 0：手动基线
   · 用户亲自掌舵。
   · 先体验无风目标阶跃，再加入恒定侧风。
   · 记录平均绝对误差、最大误差、操作次数、误差过零次数和控制能量。

2. 阶段 1：P
   · uP = Kp·e
   · δcmd = clamp(uP, ±35°)
   · 提供 Kp 滑块和过小、平衡、过大预设。
   · 展示 Kp 过小响应慢、Kp 过大可能摆动。
   · 在持续扰动下突出稳态误差、P 输出和抵消扰动所需舵量。
   · 严谨说明：并非所有 P 控制系统必然有稳态误差；在本模型的恒定扰动条件下，P 需要非零误差产生持续补偿舵量。

3. 阶段 2：PI
   · uP = Kp·e
   · uI = Ki·∫e dt
   · δcmd = clamp(uP + uI, ±35°)
   · 使用与 P 完全相同的扰动。
   · 使用积分容器或记忆槽显示误差累计。
   · 说明误差接近零后 I 项仍能维持抵消扰动所需的舵量。
   · 展示 Ki 过大导致的超调、振荡和积分饱和。
   · 实现可切换 Anti-windup。

4. 阶段 3：PID
   · uP = Kp·e
   · uI = Ki·∫e dt
   · uD = Kd·de/dt，或使用测量值微分。
   · δcmd = clamp(uP + uI + uD, ±35°)
   · D 项必须经过可调低通滤波。
   · 可切换显示原始微分和滤波微分。
   · 展示 D 项如何降低超调、振荡和调节时间。
   · 展示 Kd 过大造成的噪声敏感和舵机抖动。

5. 每个阶段包含：
   · 实验前预测问题。
   · 当前目标。
   · 实时提示。
   · 参数调节。
   · 开始、暂停、单步、倍速和重置。
   · 结果指标。
   · 正确解释。
   · 常见误区。
   · 下一阶段入口。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【八、教学引导模式与最优参考船，最高优先级】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

教学模式中加入可随时开启或关闭的“引导模式 Guided Mode”。

1. 引导模式目标
   · 帮助学习者理解自己当前参数与推荐参数的区别。
   · 推荐参数不是固定常数，而应根据当前船速、风力、洋流、目标阶跃大小和船舶模型动态选择。
   · 推荐值应来自预先离线搜索、运行时小规模参数搜索、规则调度 Gain Scheduling 或可靠的参数表插值。
   · 不得使用完全随机或没有依据的所谓“最优参数”。

2. 推荐参数显示
   · 实时显示：
     - 当前用户 Kp、Ki、Kd
     - 当前推荐 Kp*、Ki*、Kd*
     - 参数差值 ΔKp、ΔKi、ΔKd
   · 对每个差值给出简短解释，例如：
     “你的 Kp 偏大，响应更快，但超调风险更高。”
     “你的 Ki 偏小，持续扰动下消除静差需要更长时间。”
     “你的 Kd 偏小，接近目标时制动不足。”
   · 显示推荐参数所优化的目标：
     - 最小 IAE
     - 最小超调
     - 最短调节时间
     - 平衡模式
   · 允许用户切换优化目标，推荐参数随目标更新。

3. 参数搜索方法
   · 推荐在独立的简化物理副本中并行模拟一组候选参数。
   · 候选参数使用与玩家完全相同的初始状态和确定性扰动。
   · 使用综合代价函数，例如：
     J = w1·IAE + w2·Overshoot + w3·SettlingTime + w4·ControlEnergy
   · 不同优化目标使用不同权重。
   · 参数搜索不能阻塞主渲染循环。
   · 可以分帧计算、使用 Web Worker、预计算表格或在环境变化后低频重新计算。
   · 明确将结果称为“当前场景下的推荐参数”或“参考最优参数”，不要声称是所有系统的全局最优解。

4. 半透明最优参考船
   · 在玩家船旁或相同起点叠加一艘半透明参考船。
   · 参考船使用推荐 PID 参数，运行独立但相同的船舶物理副本。
   · 参考船与玩家船拥有：
     - 相同初始航向
     - 相同目标航向
     - 相同船速
     - 相同确定性风力
     - 相同洋流
     - 相同舵机约束
   · 参考船材质半透明，建议使用青绿色或淡金色轮廓。
   · 参考船具有柔和发光边缘或轮廓线，但不能遮挡玩家船。
   · 参考船旁显示“参考 PID”标签。
   · 玩家船与参考船的航迹使用不同颜色。
   · 小地图同时显示两艘船的航迹。
   · 实时显示双方：
     - 航向误差
     - 舵角
     - 偏航角速度
     - IAE
     - 超调
     - 调节状态

5. 空间显示方式
   · 默认参考船与玩家船从同一起点出发，但为了避免完全重叠，可采用轻微横向视觉偏移。
   · 该偏移只能用于显示，不得改变参考船的物理航迹和误差计算。
   · 提供“重合幽灵”“左右并排”“仅显示航迹”三种查看方式。

6. 自行控制时的引导
   · 在手动控制阶段，引导模式可显示一艘由参考 PID 控制的半透明船。
   · 学习者可比较自己的转舵时机与参考 PID 的舵角变化。
   · 永久船舵旁同时显示：
     - 我的舵角
     - 参考舵角
   · 当用户明显回舵过晚、舵量过大或反向制动不足时，给出提示，但不得强制替用户操作。

7. 自动控制参数实验时的引导
   · 玩家船使用学习者设定的 PID 参数。
   · 参考船使用推荐 PID 参数。
   · 两艘船同时运行。
   · 参数变化后，可选择立即重置对比实验，避免不同历史状态造成不公平比较。
   · 引导面板需要说明当前差异来自哪个控制分量。

8. 引导模式 UI
   · 提供清晰的开关。
   · 提供推荐参数卡片。
   · 提供优化目标选择。
   · 提供参考船显示方式选择。
   · 提供“应用推荐参数”按钮。
   · 提供“恢复我的参数”按钮。
   · 应用推荐参数前保存当前用户参数，方便对比和撤销。

1. 分量合成图

   · D 使用青色。
   · 总输出使用白色。
   · 饱和后指令使用黄色。

   · 目标航向

   · Rise Time
   · Overshoot
   · Settling Time
   · Steady-State Error
   · IAE
   · Control Energy

   · P、PI、PID 历史曲线叠加。
   · 玩家船与参考船实时曲线叠加。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【十、界面重新设计与防重叠要求，强制要求】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 使用明确的布局系统
   · 使用 CSS Grid 和 Flexbox。
   · 不要依赖大量固定绝对坐标堆叠所有面板。
   · 主视口、顶部栏、侧栏、舵轮区和底部数据区必须拥有明确安全区域。
   · 所有浮层设置合理 z-index 层级。

2. 桌面布局建议
   · 顶部：模式、时段、视角和全局控制。
   · 左侧：教学步骤或挑战任务。
   · 右侧：参数、引导模式和推荐参数。
   · 底部：响应图、因果链和状态数据。
   · 主视口中央尽量保持无遮挡。

3. 防止文字重叠
   · 所有文本容器必须设置合理 min-width、max-width、line-height 和 overflow 规则。
   · 禁止把重要文字依赖 text-overflow: ellipsis 隐藏。
   · 长文本应自动换行或进入可滚动区域。
   · 数值使用等宽字体并预留最大字符宽度。
   · 按钮文字不得挤压到两行后覆盖相邻控件。
   · 中英文混排时保持足够行高。
   · 任何分辨率下不得出现文字相互覆盖。

4. 响应式断点
   · 至少检查 1920×1080、1440×900、1366×768、1024×768、768×1024 和 390×844。
   · 窄屏下侧栏改为抽屉或分页。
   · 底部图表可以折叠。
   · 舵轮缩放但保持永久可见。
   · 小地图、参数面板和教学卡片不得互相覆盖。

5. 可访问性
   · 所有按钮有清晰标签和焦点样式。
   · 颜色之外同时使用文字、图标或线型表达含义。
   · 提供减少镜头晃动和静音选项。
   · 主要文字保证足够对比度。

6. 运行时布局检测
   · 可在开发代码中加入可关闭的 UI 边界检查。
   · 使用 ResizeObserver 在布局变化时重新计算 Canvas 和图表尺寸。
   · 模式切换、窗口缩放和面板展开后必须重新布局。

【十一、视角与交互】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 第三人称
   · 摄像机位于船后上方并平滑跟随。
   · 鼠标右键环绕。
   · 滚轮有限缩放。
   · 镜头不得穿入船体或海面。

2. 第一人称
   · 位于船桥或甲板前方。
   · 能看到流线型船头、船桥前景和海面。
   · 船体运动反馈克制且连贯。

3. 视角切换
   · 使用 0.5~0.8 秒平滑过渡。
   · 不得瞬移或突然改变方向。

4. 键盘
   · A/D 或左右方向键转舵。
   · S 或空格回舵。
   · V 切换视角。
   · P 暂停教学实验。
   · R 重置。

5. 触控
   · 支持拖拽舵轮。
   · 主要控件有足够点击区域。

【十四、验收标准】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

视觉：
✓ 主要船体和环境不再具有明显方块拼接感。
✓ 船体轮廓流线、平滑且细节丰富。
✓ 海洋具有大浪、细浪、反射、高光、泡沫和连续尾迹。
✓ 晨曦、正午、夕阳具有明显且精美的差异。
✓ 后期效果提升画面但不影响 UI。

界面：
✓ 所有指定分辨率下无文字重叠、控件覆盖和面板越界。
✓ 主视口、教学区、参数区、图表和状态区层级清晰。
✓ 文本可读，不依赖截断隐藏关键信息。
✓ 模式和面板切换后布局正确。

船舵：
✓ 可自行控制时舵轮永久显示。
✓ 舵轮旁明确显示指令角和实际角。
✓ 键盘、鼠标和触控均能操作。
✓ 自动舵接管时有清晰标注。

状态教学：
✓ 状态栏能显示控制到状态变化的完整因果链。
✓ 能判断误差在接近还是远离目标。
✓ 能看出舵机追随、船体延迟、惯性延续和反向制动。
✓ P/I/D 提示与实际数据一致。

PID 课程：
✓ Manual、P、PI、PID 四阶段均可实际运行。
✓ P 阶段能展示当前误差与比例输出的直接关系。
✓ PI 阶段能展示积分记忆和稳态误差消除。
✓ PID 阶段能展示微分阻尼、超调改善和噪声敏感性。
✓ 相同扰动可重复运行并公平对比。

引导模式：
✓ 实时给出当前场景下有依据的推荐 PID 参数。
✓ 能解释用户参数与推荐参数的差异。
✓ 半透明参考船使用独立物理状态和推荐参数运行。
✓ 玩家船和参考船受到完全相同的目标与扰动。
✓ 主场景、小地图、状态栏和曲线均能比较两艘船。
✓ 可应用推荐参数并恢复用户参数。

工程：
✓ 最终只有一个可运行 HTML 文件。
✓ 固定步长物理稳定。
✓ 无外部本地资源依赖。
✓ WebGL 不可用时显示明确错误。
✓ 长时间运行无明显内存持续增长。
✓ 桌面端目标 60 FPS，并具备画质降级能力。
